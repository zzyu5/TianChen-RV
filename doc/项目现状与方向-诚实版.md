## 0. 一句话定位（不变）

high-level MLIR 之后的**能力驱动 RISC-V 执行层**，对标 **Triton 偏后端**（拿**给定 op+layout** 自动生成该硬件最优码），不是 TVM。
走 **EmitC**：tcrv MLIR → EmitC → `mlir-translate` → C（`__riscv_*` intrinsic / IME asm）→ 板子成熟编译器（clang18 / SpacemiT-GCC）做**寄存器分配+指令调度**。
**这条边界是有意的**：IME 无上游 LLVM 后端；llama.cpp 是 C。**RA/调度交 clang/gcc**——我们能拿的赢只能是"自动综合出比 ggml 手写更优的 intrinsic 结构"，不是寄存器魔法。

## 1. 三条 Novelty 现状（N1 含证据分级）

### N1 —— 诚实回答你的质疑："凭什么算 novelty"

**质疑成立**：把能力建成可查询对象本身——LLVM `-mattr`+TTI+TableGen predicate 早做了，**单看主要是工程**。N1 真正立得住的只有：**同一带关系(provides/implies/conflicts)、硬件探针验证的 capability descriptor 被一个非-RVV family(IME)复用**（`IMEExtensionPlugin.cpp:185-202`，注释:195 自称 "the N1 capability hook"）。**抽掉跨 family 复用 N1 就塌** → **N1=substrate(地基)，由 N2 证明、N3 兑现，别单卖。**

**"VLEN 翻转是不是 pass"——不是 transform pass**：是 enumerate+capability-prune+argmin+stamp（stamping pass 写 attr，非重写 IR，`RVVQ40ScheduleMaterialization.cpp:12-20`）+ 一条独立 match-time legality GATE（`RVVToEmitC.cpp:1788-1824`）。

**N1 三根轴，证据强度不等（必须分开报）**：
| 轴 | 是什么 | 证据强度（诚实，v3 已纠 v2） |
|---|---|---|
| (a) **legality**：RVV0.7 vs 1.0（ta/ma policy、LMUL grid，`RVVCapabilityProfile.cpp:224-241`） | 同 kernel legality 分化 | 真两 profile（**RVV0.7=C920/Fedora**、RVV1.0=SG2044/k1）；**串行隔离封印、非 live side-by-side；主靠 lit 双-profile divergence + Fedora RVV0.7 隔离 bit-exact（e2e seal 仍 open）** |
| (b) **codebook VLEN 锚** | gather 合法性翻转 → emit 不同 LMUL | **board-SEALED on X60 硅片、objdump 两 VLEN 双证、byte-exact 0.000e+00**（iq4_nl/FP4：VLEN256→mf2 / VLEN128→m1，commit e65edf76；iq2_xxs i64-gather **反向** VLEN256→m1 / VLEN128→m2，commit 2a33474f——**flip 方向随 element-width**）。**这是 N1 最强的经验性硅片证据，v2 错标 board-pending** |
| (c) **LMUL on live RVV1.0** | argmin 选 LMUL 档 | **在唯一活跃 RVV1.0 板上 single-valued（GEVM 恒 mf2，m1 是手放 .inc-swap）→ argmin 区分力没被行使**（[[repack-winA-always-mf2]]） |

> N1 两条互补证据：**(IR 层)** legality GATE 读 `supported_sew/lmul/rvv_version`+`notifyMatchFailure`（最干净的 IR 层能力查询，非 march 字符串）；**(硅片层)** codebook anchor-flip 同一能力 fact 在两真 VLEN objdump 翻出不同 LMUL 且 byte-exact。两者并列，不互相 demote。

### N2 —— 结构成立、perf 空
零-core-branch 第二 family：IME 6 leaf MAC（K1 硅片 16/16 bit-exact），common 走 interface `dyn_cast` 非 family 名（`RVVScheduleDescriptorRegistry.cpp:471-475`）。✅ 结构。⬜ **没生成完整 IME GEMM**（完整 GEMM 是 ggml-spacemit 的；表4 5.66–12.9× mis-attributed）。

### N3 —— 最弱
见 §3。诚实：clean 的 e2e 加速来自前端 repack 选布局（§4），gearbox cost 是 capability-blind。"N3 已被硅片 win 兑现"是夸大。

## 3. tune 怎么工作（代码）+ 诚实：live 路是 memoization

三步（`RVVGearboxSchedule.h`）：① **枚举** 3 写死旋钮 `integer_core_lmul × multi_block_factor{1,2,4}(:2638) × strip_elision{robust,elided}(:2639)` = **≤12（2-anchor kernel 如 q4_0）/ 结构上界 18**（`enumerateBlockDotShapeCandidates:2633` 返回 `SmallVector<...,18>`）；② **剪枝** `VLMAX≥blockLen`（elision-cover legality `:2694-2695`）；vreg budget 恒 32=死的（值 `:771`、budget legality `:2709-2710`、注释 `:2625-2626` "never binds but the prune is genuine"）；③ **选**（`selectGenericSchedule:3563`）：

> **关键诚实**：选择**先查离线实测记录**（`lookupGenericTuningRecord→revalidate`，`:3572-3590`），有记录就用；没记录才落 capability-blind 静态 argmin（`computeBlockDotShapeCostCore:2472`，无 VLEN 参数）。**所以 live 路上 Gearbox 是"离线实测的 memoization 层"，不是主动能力感知搜索；"枚举+argmin" 只在冷启动成立。**

判定：tune = 封闭 3 旋钮 ≤12/18 候选里挑对（live 路查离线表），**发明不出 ggml 没写过的形状**。"12 个太弱、算不上 tune"成立。**扩旋钮空间=公认方向**（§6 L4）。Win-A 旋钮开/关只 1.0–1.43×（非 2–5×）。

## 4. 洞4 结案：不再当洞（你是对的）

**code-verified**：后端/kernel 层**没有一个 op 是"vs ggml 的新算法"**——`.td` 自陈 `GgmlRepackGemmQ40Q80Op`(`:4232`) "Records the **COMPLETE ggml** q4_0 16x1-REPACKED GEMM ... `ggml_gemm_q4_0_16x1_q8_0`"。

洞4 拆两层，都不是后端 novelty 的洞：
- **选 repack 这个布局** = 前端（option-2，TTGIR 前），改的是交给后端的 layout 本身。你的类比对。
- **给定布局生成那个化解 reduction 墙的快码** = 后端，**是我们编译器产物**（`emitted-repack-gemv.cpp` 编译器生成 C + 33 行手写 ABI adapter + `.inc`-swap，ENGAGED、≈手写 97%）。**本质=同类型替换**：emit 一个和 ggml 同算法、性能等同的 kernel 顶替它。

**结论**：洞4=前端选布局 + 后端同类型替换。**是我们编译器在干活、但非后端算法 novelty。撤销洞4。**

（配套：decode 卡 per-block reduction 延迟链，非带宽墙；repack 化解它。）

## 5. 实验指标框架（重构 + 红队补两个结构缺陷）

**前提纠正**：后端层无"新算法"，**自查表表2 标题"自创不同算法"错**——算法改变在前端（表5 option-2）。

**每 kernel = 我们的 MLIR→编译成 C，报三类，Win-A/Win-B 各含 micro+e2e**：

1. **正确性（gate）**：byte-exact vs 独立 scalar oracle 且 vs ggml 出厂 kernel；e2e coherence。几乎全绿。
2. **Win-A = 自消融**（旋钮 ON/OFF，同 op 同字节）。micro：q4_K 1.26/1.30×、q8_0 GEVM k1 1.95×… e2e：q8_0 decode 1.42×、q4_K 1.22×(regime-caveated)。**结构注**：RVV1.0 GEVM 恒 mf2 → **所有 GEVM 的 Win-A·rvv = N/A-by-construction，非空格**。
3. **Win-B = vs ggml 出厂 kernel**。ggml 出厂有**两个** kernel（block-dot + 它自己的 repack），**必须拆正交两列**：
   - **Win-B1 = vs ggml block-dot**（"补的布局/算法好不好"）。**关键：在 VLEN128 上 block-dot 就是 ggml 出厂的真 decode 基线**——ggml 把 16x1 repack gate 到 VLEN≥256，在 VLEN128 `case 128: break // TODO` 退回 block-dot，**根本不出 q4_0 repack**（已查 ggml 源）。**所以这不是人造弱基线**。
     - **(novelty 轴)** 选 16x1 repack 这布局=ggml 自己的算法，**选它是前端/option-2，不是后端算法 novelty**（保留 demote，正确）。
     - **(加速轴)** q4_0 repack **e2e 2.60× decode / 5.68× prefill 是真·gap-fill 加速**：ggml 在 VLEN128 没这算法，我们要打开它留作 TODO 的 repack-at-load 路、把它的真出厂 block-dot 顶替掉。**两个加速机理不同**：decode 2.60× 是 memory-locality（连续 16-块 lane 在 RAM 压力下流得更好，1.22× compute→2.6× e2e）；prefill 5.68× 是 **compute-bound**（6.36× micro≈5.68× e2e，nibble-decode 在 M 列上摊销）——prefill 传导是因为它本就 compute-bound，**不是**因为 repack 改了内存行为。两者都是真 gap-fill。
   - **Win-B2 = vs ggml 自己的 repack**（"我们 codegen 有无竞争力"，仅在 ggml 有 repack 的板上存在）：q4_0 GEMM k1 **0.997× TIE**、q8_0 GEVM k1 **0.99–1.01× parity**。**凡测过的都是 parity** = "自动追平专家手写"，主张成立非失败。
   - **对比锐化**：q4_0 是 **ggml-在-VLEN128-缺席** 的稀有 gap-fill；q8_0 GEVM 1.42×/parity（注5）+ q8_0 block-dot 1.19× edge（注19）是 **same-shape-as-ggml 的 lean-kernel parity+小 edge**（ggml 有对手）。**两类别勿用"弱基线"一把刷子抹平。**

> **红队补的两个结构缺陷（必进表）**：
> **(I) 每格加"证据状态"轴 `{measured | presumed | board-pending | N/A-by-construction}`**——否则"推定 parity"让没测格子默认算成功（老 over-optimism）。
> **(II) Win-B 拆 B1/B2**（见上）——消除"同 op 两个对手混用同一列"的基线错配。

### GEMV/GEMM 覆盖审计（code-verified）
op：GEMM `{Q40,Q41,Q4K}`=3；GEVM `{Q40,Q41,Q4K,Q50,Q80}`=5（`RVVOps.td`）。
- **(A) 注册表空洞**：**无 Q80Q80 GEMM、无 Q50Q80 GEMM** → q8_0/q5_0 **prefill 无 op 可测**（不是没测，是没造）。
- **(B) 基线错配**（最重要 catch）：表2 Win-B·rvv 全 vs block-dot（大数字），Win-B·k1 vs ggml 自己 repack（parity）→ 用 B1/B2 拆。
- **(C)** GEVM 的 Win-A·rvv 整列 N/A-by-construction（表留白=误读成没做）。
- **(D)** decode micro-vs-ggml 在 VLEN256/k1 **部分**是洞：**Q4K block-dot k1 已填 0.72×**（vs `_vl256`，表1）；只有 q4_K **repack** k1 是 oracle-only（注17）；Q40/Q80 k1-micro 仍空。
- **(E)** 表4 IME GEMM 5.66–12.9× **mis-attributed**（是 ggml-spacemit 的）。

### 待重测清单（成熟化后必测，别现在下结论）
1. **"内存墙"真判官 8B@VLEN128 on rvv 还没跑（open, runnable——板在线，v2 错写"掉线 blocked"）**，最干净的可跑重测目标。
2. q4_K Win-A·e2e regime-dependent（出厂 VLEN256 repack 成 GEMM 绕过该 kernel→对出厂 decode 影响=0），别当出厂赢。
3. "推定"格子（q8_0/q4_K GEVM e2e·rvv、q4_K GEMM e2e·rvv）是猜，必须实测。
4. ~24 block-dot 的 e2e 只测了 q4_K，其余未测、**不准默认 NULL**。
5. q4_0 GEMM prefill 5.68× 的 compiler-emitted 复测**未盖章**。
6. IME e2e 只在 decode-heavy regime 测过（≈1.0），prefill-heavy 该重测再下"净贡献=0"。

## 6. 洞3 详细解法：emitter → 成熟 compiler（停下来要搞清的重点）

### 6.1 现状（code-grounded）
emitter 是**一个 conversion pattern `VariantToEmitCFunc`** 拆 9 文件 28267 行。智能在 op-identity 派发（**~26 个 `*BlockDot*Op` ODS op / 44 个 `Ggml*Op` 总计**，一 quant 一个），**但每个 block-dot kernel 把骨架从头手写**：
- `RVVToEmitCSupport.cpp`(623L)=**名字 mangler 不发射**（21 个 `riscv*Name` + 0 `CallOpaqueOp`）。
- 发射惯用法 `VerbatimOp(stepComment)+CallOpaqueOp` **手写 732 次**。
- **~378 个裸 `__riscv_*` 字面量** + 大量 Twine 拼接；**vwmul/vwmacc/vwredsum 等宽化算指令零 mangler**，每处手拼。
- widening-chain 逐函数重推**还分叉**（`KQuant.cpp:685-688` 只 m1-base vs `:1596-1599` 加 m2-base = **潜伏 bug seam，只是今天 input 只 {mf2,m1} 故未触发**）。

### 6.2 分层重构（L0→L4）
| 层 | 干什么 | gate |
|---|---|---|
| **L0 发射词汇表** | `emitVCall`(收 Verbatim+CallOpaque)+12 缺的算指令 mangler | byte-exact |
| **L1 widening-chain deriver** | 一个纯 `deriveWideningChain` 单一真值源（顺手消潜伏 bug） | byte-exact |
| **L2 TargetInfo 读句柄** | emitter 侧 `deriveBlockDotFacts`（legality 权威仍在 Plugin 上游） | byte-exact |
| **L3 per-op 可组合 pattern 表** | 600 行 dyn_cast + 散落 `if(isXBody)` → `populateRVV<Family>Emitters`（recognizer 非互斥、顺序 load-bearing） | byte-exact |
| **L4（可选）能力驱动形状综合** | 一次能力变化重塑所有 quant 的 strip/unroll/accumulator | **不再 byte-exact**：新 lit + ggml 数值 oracle + micro/e2e 分开 |

**第一块砖**：L0a `emitVCall` + KQuant canary（用 Support 0 次的最硬整数 block-dot 类）。
**EmitC 边界**：CAN=统一形状综合/一 op 展开整 kernel/可审计；CANNOT=RA/指令调度/async-copy/warp/typed-LLVM-SSA builder（都 clang 的，不吹 out-schedule）。

### 6.3 红队的硬话（必须摆前面）
> **L0–L3 是 byte-exact → 按定义不改任何可测能力/性能 = 基建欠债清理 ≠ 成熟。** 价值只在 L4 真出货或 N2 真长出第二 family kernel 时兑现。
>
> **"成熟 compiler" headline 整个押在 L4 上**，而 L4 可选、未设计、不享 byte-exact 安全网、坐在两颗雷上（Win-C structural-NULL [[winc-structural-null]] + kernel-wins-不传导 [[kernel-wins-dont-transplant-to-e2e]]）。
>
> **最大 miss**：整理的是**发射**（本来就 byte-exact-正确），**没碰 selection/cost-model 质量**——cost model 才是自诊断真 gap。**决策质量 > 发射整洁。**
>
> **Triton cargo-culting 警告**：照搬形(populate/convert_layout/num_stages)≠拿到实(多真后端+真 cost model+真调度器)。我们~1.5 后端、1-bit cost、clang 管调度。

**洞3 诚实解法=两件事**：(必要不充分) 发射层整理 L0–L3 去玩具感；**(才是成熟度) L4 + cost-model + IME-GEMM**，带 structural-NULL/decode-washout 风险，必板上 micro/e2e 分开实测。

## 7. 科研指标 vs 实验指标（收口）
- **科研指标（结构主张）**：N1=substrate（2 真轴+codebook 硅片轴，证据分级见§1）；N2=零-core-branch 第二 family（结构✅/完整 kernel⬜）；N3=能力驱动选择+追平专家（机制✅薄/系统性 beat⬜）。
- **实验指标**：正确性 + Win-A(自消融,micro/e2e) + Win-B1/B2(vs ggml block-dot / 自己 repack,micro/e2e)，**每格带证据状态标**。
- **核心思路**：科研指标卖**能力驱动结构**；实验指标只需证"没付速度代价(parity)+偶尔更快"。**把 beat-ggml 当主线→越做越像调参玩具**；结构当主线、数字当证据，方向才清楚。

## 9. 其他构想：愿景 vs 实现程度（前 8 节没覆盖/欠 credit 的，code-grounded）

> 分级 `none / designed / scaffold / partial / done`。凡 scaffold/none 带 qualifier，不引入新 over-claim。

### 9.1 Plugin 父类家族 —— 抽象基类 BUILT，但"扁平单基类"非"家族继承树"
**愿景**（interfaces-and-registry.md / plugin-protocol/index.md / ime-plugin.md）：通用 plugin 协议让 core 不硬编码任何 family 即可编排；IME 证明零-core-branch 加 family。
**实现 = done(结构机制)/partial(IME dialect 富度)**：
- 真父抽象 `ExtensionPlugin`（`ExtensionPlugin.h:653-693`）~15 虚 hook，默认 fail-closed。**7 个 family 全 `final : public ExtensionPlugin` 直接派生**（RVV/IME/Scalar/Offload/Template/Toy/TensorExtLite）。
- 表驱动注册（`kBuiltinExtensionBundles[]` `BuiltinExtensionPlugins.cpp:34-42` + `kBuiltinBackendEmitters[]` 自陈 "future family is a ONE-LINE add"）。
- dispatch 靠 capability/identity（`lookupPlugin(origin)` 6 处；IME gate 在 fact `spacemit.ime`），core 无 `hasIME`/`=="ime"`。
- IME 真第二 family：`tcrv_ime` 6 ODS op + plugin 858L + dialect 438L + emitter 730L + 6 lit；K1 16/16 bit-exact。
**差距（三 qualifier）**：① **扁平单基类非多级家族树**（只一个父，7 sibling 直接派生，无中间 family base；IME↔RVV 关系是能力属性串非类继承）；② 零-core-branch = "**2 core 行 + ~2484 family LOC**"，非 free（忠于 spec "local work not zero work"）；③ **IME dialect MMA-only**（6 op，fragment 用整数 attr），spec 设想的 `!tcrv.ime.frag<>` type + load_frag/pack/dot/… **没 built**。

### 9.2 跨类型通用 pass / 快速加 kernel —— 一条轴真通用，承重 codegen 仍逐 kernel 手写
- **(1) TUNING/SCHEDULE 轴 = 真跨类型 DONE**：`RVVUnifiedScheduleMaterialization` 一个 walk-all pass，`module.walk + dyn_cast<TunableScheduleOpInterface>`（`RVVScheduleDescriptorRegistry.cpp:471-476`）自动发现，同一 `selectGenericSchedule+stampRVVSchedule` loop 覆盖 ~14 kernel key。新 tunable op adopt interface 即被接走、不需新 pass。**这是 §3 欠 credit 的已交付构件**（`TunableScheduleOpInterface`+`GenericScheduleCandidate`，06-17 把 6 clone pass 折成 1）。CAVEAT：`lookupRVVScheduleDescriptor`(:150-442) 是 per-key if-chain，候选 DATA 仍 per-kernel。
- **(2) 大 kernel EMISSION 轴 = PER-KERNEL 手写**：`emitScopeForLoop` 只在小-op 粒度通用；真 quant kernel 路由到专门手写 emitter（`RVVToEmitCKQuant.cpp` ~3000 行 Q6K/Q4K/Q5K/Q2K/Q3K 各一段）。**~26 个 BlockDot ODS op，一 quant 一个。**
- **(3) option-2 抽象 contraction = SCAFFOLD ONLY**：`selectContractionAlgorithm`(:66-91) branch-free 但只 3 类型，stage B byte-identical（非行为），**抽象 op 无真 producer**（`RVVLowerQuantContraction.cpp:42-46` 自陈 "NO real producer ... ONLY in lit fixtures ... NEVER in the real llama.cpp pipeline"）。
- **Track B "快速加" = 5 rung，1+1+3**：rung 1-2(Reduction/Dequant)**真自动构造** body 结构，但只针对**合成 generic int8 dot**（非真 ggml per-block kernel）；rung 3-5(Q40/IQ4NL/Q4K)只**识别签名+搭脚手架、body 委托回手写 emitter**（`RVVQ4KBlockDotSourceFrontDoor.cpp:18-22` "does NOT hand-roll any of it"）= recognition+delegation。
**差距**：通用的是 selection/scheduling + loop/ABI 脚手架；**kernel 算术仍 per-type 手写**。加"仅 tunable" kernel 便宜；加"真正新" kernel 仍需 新 ODS op + 整套手写 emitter（memory 自陈 q4_K "copy-then-adapt"、multi-day、timed out 2×）。

### 9.3 option-2 做成真机制？—— 验过了，**不升格、留前端**（含 Triton 类比定案）

用户问：Triton 有没有"低性能→高性能"的模块，若有 option-2 能不能算机制。**去 Triton 源码/文档 + 红队验了：有，但我们的 repack 不是那个东西。**

- **Triton 有 naive→fast 模块**：layout assignment（Coalesce/AccelerateMatmul/OptimizeDotOperands/RemoveLayoutConversions）+ 软件流水（Pipeliner）。**但严格 within-kernel**——只决定 kernel 已声明 tensor 怎么分布到寄存器/lane/shared，`convert_layout` 只重排寄存器/shared 值，**从不碰全局/磁盘权重存储**（官方 doc + DeepWiki pass survey 明证："None of these passes modify how input data is physically stored in global memory"）。Triton = kernel-adapts-to-data。
- **我们 repack = 反方向**：物理重排权重存储（plain stride-18→x16 stride-288、^0x88、16-way 交织，`RVVLowerQuantContraction.cpp:263-269`）= data-adapts-to-kernel。**"repack = Triton layout assignment 的 RISC-V 移植" 假，范畴不同。** 真对应物 = **离线 prepack 工具**（Marlin/AWQ/GPTQ、CUTLASS prepacked-B、`torch._convert_weight_to_int4pack`），**GPU 世界故意留在编译器外**。
- **"搬进编译器=novelty/第三类(compiler-DRIVEN harness-EXECUTED)" 升格，塌了**：(a) `selectContractionAlgorithm` = 算法路径选择 = 已 demote 前端；(b) materialization 与 ggml **冗余**（ggml load 时自己做 plain→x16，我们 packer 是它 byte-identical 复制品 memcmp==0）、**只活 lit**（抽象 op 无真 producer，grep `create<GgmlQuantContractionOp>`=空）、**perf-null**。拥有框架已出厂转换器的复制品 ≠ 占据空白设计点。"contract" 是 enforce 不了的 inert string attr（`:281` 声明、`:248` 仍 forward plain 指针）。
- **不赢**：唯一 e2e cell（q4_0 decode C920-VLEN128 2.6×）= 既有 STORED-x16 kernel + ggml load-time pack，选择/物化零贡献；K1-VLEN256 decline 去掉**我们自己**的 0.74× 烂 kernel = 匹配 ggml 非 beat。"3-fact AND" 是 1-bit cost model 装扮的（2/3 fact 是硬编码 per-quant lookup，只 minVLEN 是 derived）。
- **升格 = relapse**：发明"第三类"正是项目反复抓到的 scaffold-inflation（[[winc-structural-null]] 同类）。

**定案（HYBRID，偏 keep-frontend）**：
1. **option-2 的"选择+物化拥有布局"不立为新 novelty/第三类**——拒绝这个发明本身就是要点。
2. **但保留真东西**：**capability-driven DECLINE**（minVLEN==256 翻 block-dot、去掉实测 0.74× regression）banked 成"前端选择 pass 靠能力驱动 regression-removal 挣到存在价值"，**不需新类、不声明 beat**。
3. Step-1 SSA-link（把 packer SSA-接进 consumer）只做成 **bounded lit IR-vocabulary artifact**（dialect 能把 select+produce+consume+contract 表达成 SSA），**明确非 production materialization**，保 M2 byte-seal。
4. **铁律护栏**："编译器自主 选+产出+消费 匹配布局" 这句**永不准脱离三让步单走**：① 抽象 op 无真 producer、只 lit 演示；② 每次真跑是 ggml 做 plain→x16、我们 packer 冗余且不被触达；③ **零新 perf**（2.6× 是既有 STORED-x16+ggml-pack）。脱离三让步单走=退役 over-claim 回归。
5. **per-tensor/per-call 硬结不溶解**：`get_optimal_repack_type` 按 TENSOR 在 load 定一次布局，M-regime（decode/prefill）是 per-CALL——一个 tensor 一个布局，**decode-decline 与 prefill-keep 在同 tensor 互斥**；contract 化只把这个限制从编译器 **relocate 到系统**（SYS-a dual-store 2× 内存 / SYS-b JIT-per-call 只 prefill 摊销不 decode / SYS-c pick-one），不消除。

**真正该长的"Triton naive→fast"在另一处**：Triton 性能来自 **within-kernel layout + 软件流水**——**我们对应的是 Win-A/Gearbox 的 LMUL/strip 选择 + L4 形状综合**（AccelerateMatmul+Pipeliner 的真类比）。**row-② 系统性 beat 的唯一来源 = 综合一个 ggml 没手写的 in-kernel 形状**（wide-LMUL/VLEN-tuned strip/multi-accumulator）。**repack 是离线-prepack（连 Triton 都不碰）；within-kernel 形状综合才是我们该自己拥有的机制。**
