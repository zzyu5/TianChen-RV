# Maturity audit: what is our Win-A / N3 "tune" actually? (read-only, no lib/ edits)

## 1. Spec's N3 novelty claim (verbatim)
- index.md N3: **"capability/resource-aware 的跨 family 调优 … Gearbox 的候选空间由 capability + resource facts 推导，且在若干 kernel 上**实测赢** scalar 与 naive RVV"**.
- generation-selection-tuning.md: candidate space "必须由编译器可见的 capability + body facts 推导，而非硬编码常量" over a RICH fact set (VLEN/ELEN/vreg budget, mask/v0, SEW/LMUL/EMUL, widening pressure, peak-live groups, memory form, tail/mask policy…); "**只有当 Gearbox 枚举并按 resource facts 剪枝候选时，它才是 resource-aware 的。固定单候选、固定 unroll、固定 LMUL 只是 MVP 占位**"; "**实测赢家是该 key 的形状 authority**" (measured > static). It DOES promise layered autotuning (Static/AOT, Offline-profile, JIT) AND cross-family reuse to IME/Offload. So the spec asks for something LARGER than "pick a schedule by capability": a measurement-authoritative, resource-fact-pruned, **cross-family** tuner.

## 2. Axes the code actually tunes
Four, all RVV-only: (a) **integer_core_lmul** {mf4,m1,(+m2 for q8_0)}; (b) **multi_block_factor** {1,2,4} = unroll; (c) **strip_elision** {robust,elided}; (d) **accumulator-LMUL rung** (source/product/accumulator/result LMUL, m1..m8) for the low-precision widening path, live-stamped in RVVContractionSelectedBodyRealizationOwner.cpp:2861 via `selectRVVLowPrecisionMaxLegalAccumulatorLMULRung` (a max-legal-by-vreg-budget prune, NOT measured; reserve=8 is hand-set, code admits "the headroom is hand-set"). `deriveRepackHalfLanes` is a deterministic `min(vlen/16, weight_interleave)` capability→config map (N1-style divergence, not a measured tune). No tile/dtype/fusion axis.

## 3. Auto-or-hand
A **real offline loop exists** (tune_block_dot.py: dump-candidates → emit C per candidate → byte-exact-gate vs ggml → time best-of-N on `ssh rvv` → write tuning_record.txt; selectGenericSchedule reads it, measured-best overrides static-argmin when still-legal). BUT: (i) the **default pipeline runs `tune-record=""`** → the static **cost-model argmin** decides; the "实测胜出 authority" only fires when a record is hand-supplied offline. (ii) The cost model is a **hand-calibrated structural formula** (kReductionUnitCost=600 etc., "MEASUREMENT-CALIBRATED … relative-ranking, not absolute-ns"). (iii) The ONLY capability input is **one bit, Zvl128b** — code says "the ONLY capability input; the cost model is capability-blind"; the spec's rich fact set is not consumed. deriveRepackHalfLanes is a fixed VLEN rule, not measured.

## 4. Honest gap judgment
It is **between** the two framings: more than "pick 1 of 2-3 hand variants" (there's a genuine enumerate→prune→byte-exact-gated→measure→select loop with a measured>static authority order), but well short of the spec's N3. Gaps: (a) **cross-family is ZERO** — every tune mechanism is RVV-only; lib/Plugin/IME has no tune/cost/measured path; the spec's "跨 family 一套机制" and IME/Offload reuse is **aspiration, unproven** (memory: IME perf=N3 OPEN). (b) **Resource-awareness is 1 capability bit (Zvl128b) + static vreg arithmetic**, not the spec's multi-fact pruning. (c) **Default config uses the static argmin**, not measurement — the headline "实测 authority" is a non-default, manually-driven mode. (d) **Baseline honesty**: spec bar = beat scalar + naive RVV; the measured records show many block-dots <1x vs **ggml** (q4_0 0.942x) and no e2e win — "beats naive RVV" (partially met on some kernels) must NOT be laundered into "beats ggml / wins e2e" (not met). Net: the tune is a real-but-narrow RVV LMUL/unroll/elision/accumulator autotuner with a working measurement loop; the N3 paper claim (cross-family, multi-resource-fact, measurement-default, measured WIN over naive RVV) is materially larger than what's built.
