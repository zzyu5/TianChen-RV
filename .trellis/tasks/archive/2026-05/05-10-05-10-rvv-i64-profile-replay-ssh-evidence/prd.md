# Prove profile-replayed i64 RVV path on ssh rvv

## Goal

Produce one bounded, sanitized evidence route proving that replayed RVV profile
facts for the finite i64m1 slice can drive the existing compiler planning and
target export path into generated RVV C-intrinsic artifacts that compile and
run on the real `ssh rvv` host. The preferred path is `i64-vadd`.

- The prior archived task `05-10-rvv-i64-profile-replay-parity` completed the
  profile/replay parity work: i64m1 capability/profile facts are first-class,
  replay tooling can preserve them, and local compiler/export tests were
  updated without making a runtime claim.
- The next required evidence is not another metadata-only helper. It must
  start from sanitized RVV probe/profile replay facts, run the existing
  C++/MLIR planning and export path, and compile/run generated artifacts on
  `ssh rvv`.

## Boundaries

- Compiler implementation stays in C++ / MLIR / LLVM / TableGen / CMake / lit
  / FileCheck.
- Python is allowed only for evidence orchestration, artifact parsing, command
  construction, sanitization, and focused self-tests.
- Python must not decide legality, selection, lowering, emission, route
  selection, runtime ABI shape, compiler correctness, or arithmetic semantics.
- The generated RVV kernel body must come from the compiler/exporter path. A
  tiny external C caller or harness may be generated by the evidence runner,
  but it must call the compiler-generated ABI and must not replace the kernel.
- RVV runtime or correctness claims require fresh real `ssh rvv` evidence for
  the exact generated artifact path.
- Evidence artifacts must be untracked and bounded under
  `artifacts/tmp/rvv_i64_profile_replay_e2e/<run-id>/`.
- No performance, throughput, generic RVV backend maturity, or broad matrix
  claim is in scope.

## Requirements

1. Add or repair a repo-owned evidence path, preferably by extending
   `scripts/rvv_microkernel_e2e.py` or a narrowly scoped sibling, that starts
   from sanitized RVV probe/profile replay input or from
   `scripts/rvv_probe_to_mlir.py --emit-target-profile --frontend-lowering
   i64-vadd`.
2. Run the existing compiler pipeline rather than bypassing it:
   replay/input MLIR -> `tcrv-opt --tcrv-execution-planning-pipeline` ->
   target-owned RVV source/header/object or bundle export.
3. Compile on `ssh rvv` using selected bounded `march`/`mabi` capability facts
   and run a deterministic small i64 vadd external caller.
4. Compare output and record success only if the remote program exits
   successfully and emits the expected bounded success marker or result.
5. Persist sanitized JSON evidence with schema/version, run id, command
   summaries without secrets, selected family, selected march/mabi, source /
   object / binary digests when available, remote status, run result, concise
   logs, and exact failure stage when unsuccessful.
6. If compiler-generated source, runtime ABI metadata, target exporter output,
   or selected capability metadata is insufficient, fix the active C++/MLIR or
   target-export boundary rather than masking the issue in Python.
7. Add focused local tests where practical for self-test/dry-run command
   construction, replay-to-export plumbing, evidence schema/sanitization, and
   failure handling without requiring live ssh in default checks.
8. Update specs only if this round establishes a durable evidence contract.
