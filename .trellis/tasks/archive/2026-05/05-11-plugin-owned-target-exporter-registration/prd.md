# Plugin-owned target artifact exporter registration through the extension pipeline

## Goal

Add a narrow C++ registration boundary that lets extension/plugin-owned target
artifact exporters populate a `TargetArtifactExporterRegistry` through the
extension pipeline, then migrate the Toy metadata artifact exporter so Toy owns
its exporter registration through that generic boundary.

## Background

Commit `d723b52` completed `toy-plugin-target-artifact-route`: selected Toy
emission-plan metadata produced by the execution-planning pipeline now reaches
`tcrv-translate --tcrv-export-target-artifact` through real
`TargetArtifactCandidate` and `TargetArtifactExporter` validation.

The remaining integration friction is that adding the Toy artifact route still
required central builtin target-exporter registration to name Toy directly.
This task should keep the real front-door artifact path, but move the Toy route
ownership behind an extension/plugin-owned target exporter registration hook.

## Requirements

- Add the smallest generic C++ interface needed for extension plugins, or an
  adjacent generic helper, to register plugin-owned target artifact exporters.
- Keep dependency direction clean: tools or target registration composition may
  connect plugin registries to target exporter registries, but shared core and
  generic target export logic must not gain Toy/RVV/scalar/offload semantic
  branches.
- Add an `ExtensionPluginRegistry` or adjacent helper that can populate a
  `TargetArtifactExporterRegistry` from enabled extension plugins or plugin
  target-support bundles.
- Move Toy metadata artifact exporter registration so Toy owns it through the
  new boundary. Central builtin target exporter aggregation may compose generic
  registration calls, but it must not contain Toy semantic validation or a Toy
  exporter body.
- Preserve existing RVV, scalar, and offload target exporter behavior. Do not
  migrate those target families unless a tiny mechanical adapter is needed for
  build coherence.
- Fail closed for duplicate exporter route ids via the generic
  `TargetArtifactExporterRegistry` path, including the new plugin-owned
  registration helper.
- Fail closed for missing or disabled plugin-owned exporter registration where
  the API can detect it.
- Preserve deterministic Toy metadata output and route-local candidate
  validation. The Toy route must still consume selected emission-plan and target
  artifact candidates generated by the existing `tcrv-translate` front door.
- Update durable specs only if the new registration boundary needs to be
  recorded; keep any spec edit concise.

## Non-Goals

- Do not implement a Toy executable backend, runtime execution, correctness
  proof, benchmark, or performance claim.
- Do not add RVV functionality, RVV evidence fixtures, or RVV
  runtime/correctness/performance claims.
- Do not add compute semantics to `tcrv.exec`.
- Do not add Toy/RVV/scalar/offload semantic branches to generic core passes or
  generic target export logic.
- Do not implement compiler internals, plugin registry, target artifact
  registration, lowering, or emission in Python.
- Do not stop at docs-only, helper-only, or registration-only scaffolding
  without Toy front-door proof.

## Acceptance Criteria

- [ ] A generic C++ plugin-owned target artifact exporter registration boundary
      exists and is wired through the extension plugin/registry or a narrow
      adjacent helper.
- [ ] Enabled extension plugins can populate a `TargetArtifactExporterRegistry`
      through the new boundary.
- [ ] Toy metadata artifact exporter registration is plugin-owned through the
      generic boundary.
- [ ] Generic core/export logic remains free of Toy semantic branches and
      `tcrv.exec` remains compute-free.
- [ ] Existing RVV, scalar, and offload exporter routes remain registered and
      behaviorally unchanged.
- [ ] Duplicate route ids fail closed through the generic registry path.
- [ ] Missing or disabled Toy plugin-owned exporter registration fails closed
      as an unknown or unavailable target artifact route before artifact output.
- [ ] Existing Toy positive route still runs through
      `tcrv-opt --tcrv-execution-planning-pipeline` and
      `tcrv-translate --tcrv-export-target-artifact`, consuming real selected
      emission-plan / target artifact candidates.
- [ ] Existing Toy runtime ABI mismatch negative test still fails closed.
- [ ] A focused C++ test proves plugin-owned exporter registration through the
      new boundary, including duplicate route fail-closed behavior if practical.
