# L0b/L0c byte-exact migration plan (6-agent sweep analysis, 2026-06-26)

> Source: workflow wlqnvrl3u over the 9 RVV emitter files. Authoritative for the emitVCall consolidation sweep. byte-exact invariant: the 111-fixture content-md5 `9fb6344d` must stay unchanged through the WHOLE sweep.

## Convertible vs tricky

CAMPAIGN TOTAL: 656 semantic call_opaque sites = 327 CONVERTIBLE (route through EXISTING emitVCall today, byte-exact by construction) / 329 TRICKY (need a richer helper). multi-result=0, getResult(N>0)=0, vset/vsetvl-REGION=0 across all five scans (those extension categories are EMPTY — no region/multi-result work in scope).

PER-FILE-GROUP:
- core+support (RVVToEmitC.cpp): 77 sites = 46 conv / 31 tricky. (Support.cpp has 1 call_opaque = the emitVCall body itself, the target, not a site.)
- kquant (RVVToEmitCKQuant.cpp): 91 = 24 conv / 67 tricky.
- blockquant+ternary (BQL 223 + TB 55 = 278): 147 conv / 131 tricky.
- codebook (CodebookFp4 46 + GridCodebook 66 = 112): 33 conv / 79 tricky-or-OOS.
- dequant+forward (DeferredDequant 27 + ForwardElementwise 71 = 98): 77 conv / 21 tricky.

TRICKY SUB-BUCKETS (drive which helper each needs):
- IMMEDIATE/LITERAL-INTERLEAVE (build LiteralOp internally; the L0a canary class) -> emitVCallBuilt. Largest bucket: core ~11, kquant 41, BQL+TB 86, codebook ~46, dequant 10. ~190+ sites.
- POINTER/ACCUMULATOR-BUILD-INSIDE (AddOp/strided/seg2/SubOp-remaining/scalar-from-buffer/accumulator Load) -> same emitVCallBuilt contract. core 11 (4 AddOp +2 strided +2 seg2 +2 scalar +1 SubOp), kquant 7 accum-load, dequant 3 setvl-remaining. ~21 sites.
- VOID STORES (TypeRange{}) -> emitVCallVoid(+Built). core 9, kquant 11, BQL+TB 9, dequant 8 (+codebook stores not separately enumerated). ~37 sites.
- RAW/CAST/LIBM CALLEE (no __riscv_ prefix) -> emitOpaqueCall. kquant 8 fp16, BQL+TB 28 cast, dequant 3 libm (sqrtf/cosf/sinf), codebook fp16+ldexpf (ldexpf OOS). ~40 sites.
- CUSTOM-COMMENT (semantic label payload) -> emitOpaqueCall w/ commentOverride. codebook 6 (4 table +2 kmask), dequant sqrtf overlaps. ~6 sites.
- NO-COMMENT-TOKEN placeholder seeds -> NOT an emitVCall variant; defer to a separate valueMap refactor CL. BQL 8 sites.

KEY CLASSIFICATION TRAP (must classify PER-SITE not per-mnemonic): identical intrinsic, opposite category, decided solely by whether the operand was built BEFORE or AFTER the verbatim. e.g. setvl is CONVERTIBLE at silu:565/softmax:689 (remaining built before the vcall) but TRICKY at F1:91/F3:315/DEQ:1034 (remaining built after the manual verbatim); CodebookFp4 vwredsum is convertible (vl is a prebuilt named Value) but GridCodebook vwredsum is interleave (trailing sizeLit inline). An auto-sweep keyed on mnemonic WILL mis-bucket these.

The 327 convertible is the byte-exact-zero-risk structural ceiling reachable with the EXISTING emitVCall; the Tier-1/Tier-2 manglers are the ergonomic enabler that lets the convert CL avoid inventing a per-site underscore split (and add suffix/type-checking), not a correctness gate.

## L0b — richer emitVCall design (emitVCallBuilt + void sibling)

GOAL: subsume the tricky categories WITHOUT changing one emitted byte. The single hard invariant the current hand-spliced code obeys at EVERY tricky site is: [1] emit the step-comment VerbatimOp FIRST, [2] build the interior ops (immediate LiteralOp / AddOp pointer / SubOp remaining / interleave Mul+Add / scalar SubscriptOp+LoadOp / accumulator LoadOp) in source order, [3] create the call_opaque LAST. Plain emitVCall breaks this because its operands are pre-evaluated as C++ args BEFORE its internal verbatim — so any LiteralOp/AddOp/SubOp built to form the ValueRange prints BEFORE the comment and reorders the dump. The fix is to defer operand construction to AFTER the verbatim.

PRIMARY: ONE callback variant (replaces the five+ specialized build-*-inside helpers the scans proposed — same ordering contract for all of them, so prove it once):
  mlir::Value emitVCallBuilt(PatternRewriter &rw, Location loc, Type resultType,
      StringRef mnemonic, StringRef suffix, StringRef opName, StringRef role,
      llvm::function_ref<SmallVector<Value>(OpBuilder &, Location)> buildOperands);
  Body (EXACT order, this IS the contract):
    std::string callee = ("__riscv_"+mnemonic+"_"+suffix).str();
    rw.create<emitc::VerbatimOp>(loc, stepComment(opName, role, callee));   // FIRST
    SmallVector<Value> ops = buildOperands(rw, loc);                        // interior, source order
    return rw.create<emitc::CallOpaqueOp>(loc, TypeRange{resultType}, callee, ops).getResult(0);
  Each tricky site moves its literal/pointer/scalar/accumulator construction INTO the lambda, verbatim->build->call order preserved by construction. This one function covers: immediate-internal (vsetvl avl, vmv_v_x/vfmv_v_f seed, vfcvt/vfmul/vfadd inline sizeLit, all _vx imm), build-AddOp-pointer (vle base+i), build-strided-pointer (vlse byte-stride), build-seg2-pointer (vlseg2e base+i*2), build-scalar-from-buffer (vmv_v_x base[0]), build-SubOp-remaining (in-loop setvl n-i), and accumulator-load (vwmacc/vfadd Load(accVar)). Whether the interior op is a LiteralOp, AddOp, SubOp or LoadOp is irrelevant — the contract is positional, not type-specific.
  CAVEAT to test, not assume: the dequant setvl-with-remaining sites (DEQ 1034, FWD 91/315) build a SubOp(n-i) that emitc MAY fold into an inline expression operand `__riscv_vsetvl(n-i)`, in which case order is byte-invisible; classify tricky, route through emitVCallBuilt, and gate byte-exact rather than presuming the reorder mattered.

SIBLING A — VOID: emitVCallVoid(rw, loc, StringRef mnemonic, StringRef suffix, ValueRange operands, StringRef opName, StringRef role) -> void. Identical to emitVCall but TypeRange{} and NO getResult(0). Pure-additive 5-liner; subsumes the pure-void stores (operands pre-built: core 4362/4826, kquant 150/1028/1155/2722/3356, dequant several). For stores that ALSO interleave (void AND a built operand: core 2189/3099/3794/3951/4107/4577/4694, kquant 6, dequant 8, BQL 9, TB 1), add emitVCallVoidBuilt with the SAME callback contract as the primary (verbatim->build->call, no result). ~37 void stores campaign-wide route through these two.

SIBLING B — RAW CALLEE + COMMENT OVERRIDE: emitOpaqueCall(rw, loc, Type resultType, StringRef fullCallee, ValueRange operands, StringRef opName, StringRef role, std::optional<StringRef> commentOverride). Passes fullCallee VERBATIM (no "__riscv_"+_+ synthesis), and when commentOverride is set, the VerbatimOp payload is stepComment(opName, role, *commentOverride) instead of the callee. Covers THREE non-manglable shapes with one helper: (a) libm seams cosf/sinf (callee=="cosf", operand pre-built — trivial) and sqrtf (callee=="sqrtf" AND comment payload is the semantic label "scale" AND meanPlusEps AddOp interleaved -> use the BUILT variant + commentOverride); (b) fp16 cast-deref reads `(float)*(const _Float16 *)` (kquant 8, BQL 24+TB 4, codebook per-method) — single pre-built addr, single float result; (c) codebook custom-comment table/kmask loads (CodebookFp4 149/652/1149/1722, GridCodebook 860/1551) whose verbatim is "codebook_table_load"/"kmask_table_load" not the callee — commentOverride preserves that byte-for-byte while the real callee is the __riscv_vle8 string. ldexpf / E8M0 / UE4M3 scale reconstruction stay OUT (L1 widening-chain, do not touch).

MULTI-RESULT / REGION: NOT NEEDED. All five scans verified zero multi-result calls, zero getResult(N>0), zero vset/vsetvl REGION call_opaque (the only regions are emitc ForOp/ExpressionOp, not calls). So emitVCall stays single-result; no region/multi-result extension is in scope. The seg2 tuple-create/load returns a single tuple Type (one Type in TypeRange) -> already fine via the primary callback.

NO-COMMENT-TOKEN: the 8 BQL placeholder seeds (`valueMap[op.getResult()] = resultTok`, vmv_v_x with NO preceding step-comment) must NOT go through any emitVCall* variant (all of them inject a comment byte). Either an emitVCallNoComment shim or, cleaner, a valueMap refactor that drops the placeholder call — a SEPARATE CL, explicitly excluded from the automated sweep.

## Risks

- ORDERING is the whole hazard for the ~210 interleave/pointer sites: emitVCallBuilt MUST create the step-comment VerbatimOp, THEN run the buildOperands lambda (interior LiteralOp/AddOp/SubOp/LoadOp in source order), THEN the call_opaque. Any helper that pre-evaluates operands (plain emitVCall) reorders the dump and fails the byte gate. Prove the contract on TernaryBinary FIRST.
- CALLEE-SPLIT typo surface: emitVCall rejoins '__riscv_'+mnemonic+'_'+suffix with exactly ONE underscore, so every clean site must split its full callee at an underscore (vwmul|vv_i32m1_m, vsetvl|e32m1, vmv_x_s|i32m1_i32). The 34 hardcoded m2 exp-polynomial strings in ForwardElementwise.cpp are the densest typo trap — prefer a FULL-callee overload of emitVCall for the vcall lambdas so no split is invented.
- standaloneReductionIntrinsicName promotion to Support: byte-neutral ONLY because the trailing 'm1' result-LMUL is invariant across every observed reduction (vredsum/vwredsum/vwredsumu/vfwredusum/vfredmax all land result in m1). Do NOT adopt the 6-arg dstLmul signature the BQL scan proposed (it would diverge). The promotion flips 5 call sites (RVVToEmitC.cpp:2251/2344 + DeferredDequant.cpp:81/563/878) from the class-static form — gate that diff alone.
- MUST-SKIP from any automated L0c sweep or bytes change: (a) the 8 BQL no-comment placeholder seeds (routing through emitVCall* injects a comment byte) -> defer to a valueMap refactor CL; (b) the 4 CodebookFp4 table-loads + 2 GridCodebook kmask-loads (custom semantic comment) -> only emitOpaqueCall-with-commentOverride; (c) ldexpf / E8M0 / UE4M3 scale reconstruction -> L1 widening-chain, untouched.
- PER-SITE classification, never per-mnemonic: same intrinsic flips convertible<->tricky purely on whether the operand was built before/after the verbatim (setvl silu:565 conv vs F1:91 tricky; vwredsum CodebookFp4 conv vs GridCodebook interleave). A mnemonic-keyed sweep mis-buckets and reorders.
- BLISTER on inline-fold uncertainty: the setvl-with-remaining SubOp(n-i) (DEQ 1034, FWD 91/315) MAY fold into an emitc inline expression operand making the reorder byte-invisible — classify tricky and byte-GATE, do not assume either way.
- BQL repack hazards: the hoisted-once vl8/four prebuilt Values must use PLAIN emitVCall (the internal-avl overload would rebuild the literal and change bytes); and the in-lambda step() comment must be dropped before routing a step()-bodied call through emitVCall or the comment doubles.
- FINGERPRINT drift: 9fb6344d is the L0a baseline/floor, not a forever-pinned target — the 111-fixture glob grows with coverage commits and absolute fingerprints have gone STALE before (project memory). Each batch gate is forced/CLEAN rebuild (ODS .inc regenerates, tcrv-opt sometimes doesn't relink) + BEFORE==AFTER content equality of the emitted C, NOT fixed-hash matching. A coverage commit landing mid-sweep will move the hash legitimately.
- void-store helper must emit TypeRange{} and NOT call getResult(0); a copy-paste that leaves getResult(0) compiles against a 0-result call and crashes/asserts rather than failing the byte gate.
- vreinterpret mangler/site arity: exactly ONE operand and NO vl — a SEW/LMUL-style mangler or helper that infers a trailing vl operand will silently add an argument and change bytes.
